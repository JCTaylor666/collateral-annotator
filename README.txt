==========================================================================
 血管分段侧支标注器 (annotator) —— software/ 目录说明
==========================================================================

用途
----
本目录存放标注器的全部前端代码。纯 HTML + JavaScript。标注器读取合成/真实数据,
让用户逐段点选"侧支",并把每个单元的标注结果保存成 json。
(能否直接写进数据文件夹取决于"保存机制",见文末。)

--------------------------------------------------------------------------
目录组成与各文件功能
--------------------------------------------------------------------------
software/
  README.txt        本文件。目录组成、各文件功能、坐标约定、数据格式约定、维护约定。
  index.html        [计划] 应用入口:页面骨架(工具栏 / 画布 / 侧栏 / 底部导航)。
  css/
    style.css       [计划] 全部样式。
  js/
    app.js          [计划] 主逻辑:全局状态、事件绑定、case/unit 切换、保存/载入。
    loader.js       [计划] 数据载入:解析所选文件夹(webkitRelativePath),按
                            case_* / (frame_*|minip) 组织成树;每单元定位其文件。
    npy.js          [计划] .npy 解析器:解析 numpy header,按 descr 生成 TypedArray
                            (<u2->Uint16Array,|u1/<u1->Uint8Array);fortran_order=true
                            或不支持的 descr 直接报错,不静默误读。
    canvas.js       [计划] 画布渲染:铺底 frames.png、选中/悬停高亮、50px 刻度尺、
                            画布坐标 <-> 图像像素换算(含 devicePixelRatio 与缩放偏移)。
    state.js        [计划] 标注状态:每个 (case, unit) 的已选段与点击坐标、撤销栈、
                            localStorage 自动缓存(只存标注,不存解码后的大数组)。
  serve.py          [计划,待定] 极简本地服务:发布应用 + 接收保存请求,把
                            annotation.json 写进对应单元目录(仅"保存机制方案 A"需要)。

  [计划] = 待功能确认后创建;新增/改名任何文件都必须在此登记。

--------------------------------------------------------------------------
坐标与索引约定(重要,避免 x/y 写反)
--------------------------------------------------------------------------
- 图像坐标:原点在左上角,x = 列(向右为正),y = 行(向下为正)。屏幕/人眼用 (x, y)。
- 数组索引:label / mask 形状为 (H, W) = (行, 列),行主序(C order)。
      W = shape[1], H = shape[0]
      查段:label[y, x]        扁平化:flat[y * W + x]
  即人眼的 (x, y) 对应 numpy 的 [y, x] —— 这就是 "x,y 还是 y,x" 的来源。内部一律用
  (x, y),索引时才换成 [y, x]。
- 点击换算:imgX = floor((clientX - rect.left - offsetX) / drawScale),y 同理;drawScale
  用绘制时同一个缩放系数,并考虑 devicePixelRatio;结果 clamp 到 [0,W-1] / [0,H-1]。
- 载入时断言 png.width==W 且 png.height==H。注意:合成数据是正方形 800x800,x/y 写反
  "看不出来";必须用一张非正方形数据做转置测试。
- GUI 画布叠加一把刻度尺:顶部标 x、左侧标 y,每 50px 一刻度、每 100px 标数字,原点左上。

--------------------------------------------------------------------------
读取的数据格式约定(annotator 只认这个结构)
--------------------------------------------------------------------------
给 annotator 一个根路径(如 synthetic_data/),它按下述规则发现单元,忽略无关文件:
  - 病例 = 名字匹配 ^case_\d+$ 的目录;按其中数字自然排序。
  - 单元 = 病例下匹配 ^frame_(\d+)$ 或 ^minip$ 的目录;frame 按数字排序,minip 排最后。
  - 每一层(根 / 病例 / 单元)都跳过以 "." 开头的项(.DS_Store 会被 Finder 反复生成)
    以及非上述模式的文件。缺文件的单元跳过并提示,不崩溃。

  synthetic_data/
    case_0001/
      frame_0/   frames.png  label.npy  mask.npy   [annotation.json]
      frame_1/   ...
      frame_2/   ...
      frame_3/   ...
      minip/     frames.png  label.npy  mask.npy   [annotation.json]
    case_0002/ ...

每个单元(frame_N 或 minip)固定三件套 + 可选的标注结果:
  frames.png       原始灰度图(8-bit,mode L)。铺在画布底层显示。血管本身可见;
                   看不见的是血管"内部的分段边界"——分段只存在于 label.npy。
  mask.npy         HxW uint8,二值血管掩膜 0/1(单标签)。可选,仅显示/校验用。
  label.npy        HxW uint16,分区图。0=背景,1..K=分段号(单元内唯一,从 1 开始;
                   单元之间可重复)。点击 -> label[y,x] = 段号 -> 切换该段。
                   K 由该文件实际唯一值决定,不要写死(minip 的 K 与各 frame 不同)。
  annotation.json  该单元的标注结果(见下节)。存在则载入续标,不存在=还没标。

  minip 单元是完整圆环(= 四帧的并集)。它的 label 是在整环上"重新分区"得到的
  全局唯一编号(1..K,本合成数据 K=12),不是把 4 帧的 label 叠加而来。

--------------------------------------------------------------------------
标注结果的保存(每个单元一个 json,存进该单元文件夹)
--------------------------------------------------------------------------
每个单元把自己的标注存成 annotation.json,放在该单元目录内,与三件套并列:

  case_0001/frame_0/annotation.json

单个文件内容:
{
  "schema_version": 1,
  "case": "case_0001",
  "unit": "frame_0",
  "image_size": [800, 800],          # [width, height]
  "coord_order": "xy",               # "xy" 或 "yx";由 GUI 切换,决定 click 的顺序
  "collaterals": [
    { "id": 2, "click": [610, 600] },
    { "id": 3, "click": [470, 660] }
  ]
}
- collaterals:该单元标为侧支的段;每个含段号 id 与"选中它时的那次点击坐标" click。
- click 按 coord_order 写:"xy"->[x,y],"yx"->[y,x];读取时按本字段反解为内部 (x,y)。
- id 应等于 label[click_y, click_x];导入时校验,对不上给警告(minip 是重算分区,
  跨数据版本 id 可能变)。
- collaterals 为空 = 看过、没侧支;文件不存在 = 还没标。
- 像素级侧支掩膜仍由 Python 侧 np.isin(label, ids) 复原;annotator 不生成像素。

保存机制(待定 —— 决定是否需要 serve.py):
  纯双击打开的 file:// 页面不能把文件写进磁盘文件夹(浏览器安全限制),要把
  annotation.json 写进各单元目录必须三选一:
  A) 极简本地 Python 服务(serve.py):既发布应用又接收保存请求写文件。跨浏览器最稳。
     代价:先跑 `python serve.py`,浏览器开 localhost(不再是双击)。[推荐]
  B) File System Access API:Chrome/Edge 专用,授权后浏览器直接写;file:// 下被禁,
     仍需从 localhost 打开;Firefox/Safari 不支持。
  C) 仅下载:保持纯双击,但只能把 json 下载到下载目录、再手动放进各文件夹。

--------------------------------------------------------------------------
维护约定
--------------------------------------------------------------------------
- 单一职责:每个 js 模块只做一件事。依赖:app 编排;loader->npy;canvas 消费(数组+png);
  state 管选择/撤销/localStorage。
- 数据格式若变化:先改本文件"坐标/数据格式约定",再改 loader.js / npy.js。
- 选择集一律以 (caseId, unitId) 为键;局部段号跨单元会重复,不可只按段号合并。
- 标注结果写进数据目录(各单元的 annotation.json);software/ 只放代码,不放数据/结果。
- 新增文件必须在“目录组成”里登记。
- 改动任何 js/css 后,把 index.html 里引用的 ?v=N 版本号整体 +1(强制刷新缓存),否则用户浏览器可能用旧缓存。
